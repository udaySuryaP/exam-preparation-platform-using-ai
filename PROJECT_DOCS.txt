  NEXT_PUBLIC_SUPABASE_URL=          Your Supabase project URL
  NEXT_PUBLIC_SUPABASE_ANON_KEY=     Supabase anon/public key
  SUPABASE_SERVICE_ROLE_KEY=         Supabase service role key (server-only)
  OPENAI_API_KEY=                    OpenAI API key (not integrated yet)
  UPSTASH_REDIS_REST_URL=            Upstash Redis REST URL
  UPSTASH_REDIS_REST_TOKEN=          Upstash Redis REST token
  NEXT_PUBLIC_APP_URL=               App base URL (e.g. http://localhost:3000)

The NEXT_PUBLIC_ prefix means the variable is available in browser code.
Variables without that prefix are server-only (never sent to browser).

SUPABASE_SERVICE_ROLE_KEY is especially sensitive — it bypasses Row Level
Security. It's only used server-side in /api/profile PUT to update auth
user metadata.

4. AUTHENTICATION SYSTEM
================================================================

Provider: Supabase Auth (email + password)
Package: @supabase/ssr (cookie-based server-side sessions)

HOW IT WORKS:

  Sign Up:
    1. User fills SignupForm (full name, email, password, confirm password)
    2. Client-side validation via Zod schema (min 8 chars password, etc.)
    3. Calls supabase.auth.signUp() with emailRedirectTo = /login
    4. Supabase sends a verification email to the user
    5. UI switches to "check your inbox" confirmation screen
    6. User clicks the link in email → lands on /login, session is active

  Login:
    1. User fills LoginForm (email, password)
    2. Calls supabase.auth.signInWithPassword()
    3. On success → redirected to /chat (or wherever middleware sends them)
    4. On failure → generic error shown (avoids leaking account existence)

  Session Management:
    - @supabase/ssr stores session in http-only cookies
    - lib/supabase/middleware.ts refreshes the session on every request
    - lib/supabase/server.ts creates a server-side Supabase client
    - lib/supabase/client.ts creates a browser-side Supabase client

  Sign Out:
    - Calls supabase.auth.signOut() from the UserProfile component
    - Middleware will redirect to /login on next request

SECURITY NOTES:
  - Error messages are generic on login/signup (no user enumeration)
  - Rate limit error shows a friendly specific message
  - Supabase errors are only logged to console, never shown raw to users

5. MIDDLEWARE & ROUTE PROTECTION
================================================================

File: middleware.ts (root) → calls lib/supabase/middleware.ts → updateSession()

The middleware runs on EVERY request before the page renders.

LOGIC FLOW:
  1. Refresh Supabase session (reads/writes cookies)
  2. Get current user via supabase.auth.getUser()
  3. If NOT logged in and route is NOT public → redirect to /login
  4. If logged in and on /login or /signup → redirect to /chat
  5. If logged in and accessing dashboard route (/chat, /courses, etc.)
     → check user_profiles.onboarding_completed
     → if false → redirect to /onboarding/step-1
  6. If logged in and on /onboarding/* but already completed onboarding
     → redirect to /chat

Public routes (no auth needed):
  - / (landing page)
  - /login
  - /signup
  - /verify-email
  - /api/courses (GET)
  - /api/patterns (GET)

All other routes require authentication.

8. CHAT SYSTEM
================================================================

Files:
  - components/chat/ChatInterface.tsx  (main controller)
  - components/chat/MessageList.tsx    (scrollable list)
  - components/chat/MessageBubble.tsx  (individual message, renders markdown)
  - components/chat/InputBox.tsx       (textarea + send button)

HOW A CONVERSATION WORKS:

  Starting a new chat:
    1. User lands on /chat (no ?id param)
    2. Empty state shown with suggested prompts
    3. User types + sends → POST /api/chat (no conversationId)
    4. API creates a new conversation row, saves user message
    5. API saves placeholder AI response
    6. Returns {answer, conversationId}
    7. Frontend adds messages to state
    8. URL updated to /chat?id={conversationId} (via window.history.replaceState)

  Loading existing chat:
    1. User clicks a conversation in RecentChats sidebar → /chat?id=xxx
    2. ChatInterface reads ?id from URL
    3. useEffect fires → fetches all messages from Supabase
    4. Shows spinner while loading
    5. Displays all messages from history

  Regenerate:
    - Finds the last user message in the array
    - Removes everything after that user message
    - Re-sends the same message

  Current AI response:
    ⚠️ AI IS A PLACEHOLDER. The response is always:
    "AI responses are not enabled yet. This feature is coming soon 🚀"
    OpenAI + RAG integration is the next major step.

URL pattern: /chat?id={uuid}  — conversationId is a UUID from Supabase

9. BACKEND API ROUTES

All routes:

---
POST /api/chat
---
Rate limit: 20 requests/minute per user

Request body:
  {
    message: string,          // max 5000 chars
    conversationId?: string,  // UUID, optional (null = new conversation)
    courseId?: string         // UUID, optional context
  }

Response:
  {
    answer: string,
    sources: [],
    conversationId: string
  }

What it does:
  1. Authenticates user
  2. Rate limits
  3. Validates message (required, non-empty, max 5000 chars)
  4. If conversationId provided → verifies it belongs to this user
  5. If no conversationId → creates new conversation (title = first 50 chars)
  6. Saves user message to messages table
  7. Fetches last 10 messages as history (for future AI context)
  8. Returns placeholder AI response (real OpenAI call is commented out)
  9. Saves AI response to messages table
  10. Updates conversation.updated_at
  11. Returns answer + conversationId

Query params:

10. RATE LIMITING (UPSTASH REDIS)
================================================================

File: lib/rate-limit.ts

WHY UPSTASH REDIS:
  The old implementation used an in-memory JavaScript Map.
  Problem: Each serverless instance has its own memory — so on Vercel
  or any multi-instance deployment, users could bypass rate limits by
  hitting different server instances. Someone on a phone and laptop
  could both send 20 req/min each.

  Upstash Redis is a shared external store — all instances read/write
  the same counters, so rate limits apply globally across all devices.

HOW IT WORKS:
  - Uses sliding window algorithm (Ratelimit.slidingWindow)
  - Redis key format: "chat:{userId}", "search:{userId}", "profile:{userId}"
  - Sliding window is fairer than fixed window (no burst at window reset)

LIMITS:
  | Route           | Max Requests | Window |
  |-----------------|-------------|--------|
  | /api/chat       | 20          | 1 min  |
  | /api/search     | 30          | 1 min  |
  | /api/profile PUT| 10          | 1 min  |

GRACEFUL DEGRADATION:
  If UPSTASH_REDIS_REST_URL or UPSTASH_REDIS_REST_TOKEN are missing
  (e.g. local dev without setup), the rate limiter warns in console
  and allows all requests. This prevents dev environment breakage.

RESPONSE WHEN RATE LIMITED:
  HTTP 429 with body: { error: "Too many requests. Please wait a moment." }
  Headers: X-RateLimit-Remaining, X-RateLimit-Reset

---
CREATE TABLE syllabus_embeddings (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  course_id  UUID NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  content    TEXT NOT NULL,          -- chunk of syllabus text
  embedding  VECTOR(1536),           -- OpenAI text-embedding-3-small output

  ON syllabus_embeddings
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

12. ROW LEVEL SECURITY (RLS)
================================================================

All tables have RLS ENABLED. Without a matching policy, all access is denied.

user_profiles:
  SELECT: auth.uid() = id          (own row only)
  INSERT: auth.uid() = id
  UPDATE: auth.uid() = id

courses:           SELECT: true (publicly readable, no auth needed)
modules:           SELECT: true (publicly readable)
syllabus_embeddings: SELECT: true (publicly readable for search)
question_patterns: SELECT: true (publicly readable)

conversations:
  SELECT: auth.uid() = user_id     (own conversations only)
  INSERT: auth.uid() = user_id
  UPDATE: auth.uid() = user_id
  DELETE: auth.uid() = user_id

messages:
  SELECT: EXISTS (SELECT 1 FROM conversations c
                  WHERE c.id = messages.conversation_id
                  AND c.user_id = auth.uid())
  INSERT: same check
  DELETE: same check
  (Users can only see messages in their own conversations)

user_progress:
  SELECT: auth.uid() = user_id
  INSERT: auth.uid() = user_id
  UPDATE: auth.uid() = user_id

NOTE: The API routes use the server-side Supabase client, which
automatically applies RLS using the user's session token.
The service role client (only used in /api/profile PUT) bypasses RLS,
which is intentional for admin operations like updating auth metadata.

13. DATABASE FUNCTIONS
================================================================

-- Vector similarity search function used by RAG:
CREATE OR REPLACE FUNCTION match_syllabus(
  query_embedding  VECTOR(1536),   -- embedding of the user's question
  match_threshold  FLOAT DEFAULT 0.7,  -- minimum similarity score
  match_count      INT DEFAULT 5,      -- number of results to return
  filter_course_id UUID DEFAULT NULL   -- optional: restrict to one course
)
RETURNS TABLE (
  id         UUID,
  content    TEXT,   -- the matching syllabus chunk
  similarity FLOAT,
  metadata   JSONB
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT
    se.id,
    se.content,
    1 - (se.embedding <=> query_embedding) AS similarity,
    se.metadata
  FROM syllabus_embeddings se
  WHERE
    (filter_course_id IS NULL OR se.course_id = filter_course_id)
    AND 1 - (se.embedding <=> query_embedding) > match_threshold
  ORDER BY se.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;

-- Auto-update trigger (fires on UPDATE of updated_at columns):
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

14. SECURITY MEASURES
================================================================

1. AUTHENTICATION
   - All API routes verify user via supabase.auth.getUser()
   - Returns 401 Unauthorized if no valid session
   - Session stored in httpOnly cookies (can't be read by JavaScript)

2. AUTHORIZATION (IDOR Prevention)
   - All DB queries are scoped to auth.uid()
   - Conversations: verified user_id matches before reading/writing
   - Messages: checked via JOIN to conversations.user_id
   - RLS provides database-level enforcement as a second layer

3. RATE LIMITING
   - Upstash Redis sliding window — shared across all devices/instances
   - Per-user keys (not per-IP) — fair and accurate
   - Graceful fallback if Redis is unavailable

4. INPUT VALIDATION
   - All API inputs type-checked (typeof checks)
   - String length limits enforced server-side
   - Integers validated as actual integers (Number.isInteger)
   - Semester validated as 1–8
   - Course IDs validated as proper UUID format via regex
   - Pagination params clamped to safe ranges (max 100 per page)

5. ERROR MESSAGE SAFETY
   - Raw Supabase errors never shown to users (user enumeration prevention)
   - Specific friendly message for rate limit errors
   - Generic message for all other auth errors
   - Real error always logged to console for debugging

6. SERVICE ROLE KEY
   - SUPABASE_SERVICE_ROLE_KEY only used in /api/profile PUT (server-side)
   - Used to update auth.users metadata via admin API
   - Never exposed to client, only in server environment
